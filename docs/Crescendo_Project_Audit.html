1. Fast Explanation You Can Memorize

One-sentence explanation: Crescendo is a React music-learning platform where students log in, choose an instrument, study lessons, take quizzes, save notes, track progress, and unlock harder lesson tiers.

Simple presentation version

Crescendo is a frontend React application built with Vite. It uses React Router v6 for pages, localStorage for accounts and progress, and reusable components for the navbar, progress bars, badges, and metronome. The app has 6 instruments, 3 tiers per instrument, 3 lessons per tier, and 4 quiz questions per lesson. That gives 54 lessons and 216 quiz questions total.

The main learning flow is: log in, choose an instrument, open a lesson, save notes, take the quiz, pass with 70%, and unlock the next tier when all lessons in the current tier are completed.

What to say if asked how the code works

  • main.jsx starts React and renders App.
  • App.jsx defines all routes and protects pages based on whether a user is logged in.
  • auth.js saves accounts and current user sessions in localStorage.
  • progress.js saves lesson completion, notes, review marks, quiz history, progress percentage, tier locks, and ranks.
  • data.js stores the curriculum: instruments, lessons, videos, diagrams, quizzes, and resources.
  • Page files in src/pages display the major screens: login, home, dashboard, lesson, quiz, progress, and resources.

4. How the App Starts

  1. The browser loads index.html.
  2. index.html loads /src/main.jsx in development, or bundled JS after production build.
  3. main.jsx calls ReactDOM.createRoot(...).render(<App />).
  4. App.jsx wraps everything in HashRouter.
  5. AppShell checks getCurrentUser() from auth.js.
  6. If a user exists, it renders Navbar, protected pages, and Metronome. If not, it redirects to /login.
index.html → main.jsx → App.jsx → HashRouter → Protected Routes → Page Components

5. Routing and Navigation

The app uses react-router-dom v6. It uses HashRouter instead of BrowserRouter because GitHub Pages can return 404 errors for direct paths like /login. Hash URLs like .../#/login load the main app first, then React Router handles the route.

RouteComponentPurpose
/loginLoginPageSign in, register, demo login.
/HomePageChoose an instrument and view standing.
/dashboard/:instrumentDashboardPageTier selection and lessons for one instrument.
/lesson/:instrument/:tier/:lessonIdLessonPageSpecific lesson loaded from data.
/quiz/:instrument/:tier/:lessonIdQuizPageSpecific lesson quiz.
/progressProgressPageProgress across all instruments.
/resourcesResourcesPageExternal learning resources.

Dynamic route example: /lesson/guitar/beginner/g-b-1 tells the app to find guitar → beginner → g-b-1 inside INSTRUMENT_DATA.

6. Authentication Flow

Authentication is frontend-only. The active React app does not call a server for login. Accounts and sessions are saved in localStorage.

Main files

  • LoginPage.jsx: form UI, validation, password visibility, register/login mode.
  • auth.js: storage logic for accounts and current user.
  • Navbar.jsx: logout button.
  • App.jsx: route guard.
Register:
LoginPage.validate()
→ auth.register({ username, email, password })
→ checks duplicate email
→ creates user with crypto.randomUUID()
→ saves account in crescendo_accounts
→ saves session in crescendo_user
→ navigate('/')

Login:
auth.login({ email, password })
→ searches crescendo_accounts
→ saves session in crescendo_user
→ navigate('/')

Demo:
loginDemo() → saves demo user → navigate('/')

Logout:
Navbar.handleLogout() → auth.logout() → remove crescendo_user → navigate('/login')

Security concern: passwords are stored in plain text in localStorage. This is acceptable only for a class frontend demo. A production app needs backend authentication, password hashing, secure sessions, and a database.

9. User Flow

  1. User opens app and lands on login if no session exists.
  2. User registers, logs in, or uses demo login.
  3. App stores current user and redirects home.
  4. User chooses an instrument.
  5. Dashboard displays Beginner/Intermediate/Advanced tiers and lesson cards.
  6. User opens a lesson, watches video, reads explanation, saves notes, or marks review.
  7. User takes quiz.
  8. If score is 70% or higher, markLessonComplete() saves completion.
  9. When all Beginner lessons pass, Intermediate unlocks; when all Intermediate pass, Advanced unlocks.
  10. Progress page and navbar rank update based on completion.

10. Feature Flow Explanations

Lesson completion and tier unlock

QuizPage.handleSubmit()
→ computes score
→ if score >= 70, markLessonComplete(instrument, tier, lessonId)
→ progress.js marks lesson true
→ checks whether all lessons in that tier are true
→ if yes, updates currentTier to next tier
→ saveProgress() writes localStorage

Saved notes

LessonPage textarea onChange → updates notes state
Save Notes button → saveLessonNotes()
progress.js writes notes under lessonNotes[lessonId]
Dashboard can show “Notes saved” metadata

Quiz history

Submit quiz → build review objects → calculate score
→ recordQuizAttempt() → stores latest attempt at top
→ keeps only 8 recent attempts per lesson

Rank

Navbar → getStudentRank()
→ getStudentProgressSummary()
→ count completed lessons across all instruments
→ show Novice, Apprentice, Performer, Virtuoso, or Maestro

12. Backend Communication

Current frontend: no API calls. No fetch(), Axios, Firebase, Supabase, or realtime backend is used by React.

Optional backend: server/server.js provides a Node HTTP API with JSON file storage, routes for health/register/login/progress, and PBKDF2 password hashing. It is not currently connected to the static GitHub Pages app.

14. Styling and UI System

The UI is styled mainly in App.css. It uses a luxury black/gold design system with CSS variables, glassmorphism cards, gold primary buttons, responsive grids, and page-specific layouts.

  • Palette: black, charcoal, metallic gold, muted cream text.
  • Typography: Playfair Display headings, Inter body.
  • Reusable classes: glass-card, btn, btn-primary, btn-ghost, eyebrow.
  • Responsive rules around 980px, 720px, 460px, and wide desktop.

Concern: App.css is 1500+ lines, so it is becoming a high-risk file.

17. Performance Notes

  • Performance is acceptable for the current project size.
  • data.js bundles all lessons into the main JS; lazy loading by instrument would scale better.
  • useMemo is used for derived values in several components.
  • Repeated progress calculations read localStorage often; a central store would be cleaner.
  • External images and YouTube iframes are the heaviest resources.

19. Refactor Recommendations

  1. Move login validation into auth.validation.js.
  2. Split data.js by instrument.
  3. Move quiz scoring into a utility.
  4. Add AuthProvider for current user state.
  5. Add useProgress() or a progress provider.
  6. Connect backend later for cross-device persistence.
  7. Add tests for tier unlock, quiz scoring, and auth validation.

20. Security Recommendations

  • Never store real passwords in localStorage.
  • Do not trust client-side progress for real grading because localStorage can be edited.
  • Use server-side validation and hashed passwords.
  • Use secure sessions/tokens and HTTPS.
  • Restrict backend CORS. Current scaffold uses *.
  • Add input validation and request size limits to the backend.

22. Final Mental Model

Think of Crescendo as three connected systems:

  1. Content system: data.js contains instruments, lessons, quizzes, diagrams, and resources.
  2. Learning state system: progress.js records what the student has done.
  3. Navigation/UI system: App.jsx and pages turn content and progress into an interactive app.
Content + Progress + Routes = Crescendo learning experience
